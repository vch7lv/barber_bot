# План разработки Telegram-бота для барбера

Документ определяет конфигурацию приложения, выбор БД и поэтапный план реализации по [requirements.md](requirements.md).

---

## 1. Конфигурация и входящие параметры

Согласно **A1**: Telegram-токен и список Telegram User ID барберов задаются при запуске; остальное — через env/конфиг/флаги.

### 1.1. Обязательные (при запуске)

| Параметр | Источник | Описание |
|----------|----------|----------|
| `TELEGRAM_BOT_TOKEN` | env | Токен бота от @BotFather. Без него приложение не стартует. |
| `BARBER_TELEGRAM_IDS` | env | Список Telegram User ID барберов через запятую, например `123456789,987654321`. Пустой список — ошибка конфигурации (должен быть хотя бы один барбер). |

### 1.2. Опциональные (env / конфиг / флаги)

| Параметр | По умолчанию | Описание |
|----------|--------------|----------|
| `DATABASE_DSN` | `file:barber_bot.db?_journal_mode=WAL` | DSN для SQLite (или URL для PostgreSQL при смене адаптера). |
| `TZ` | `Europe/Moscow` | Часовой пояс (N5); фактически фиксируем МСК в коде, env для явности. |
| `BACKUP_TIME` | `22:00` | Время ежедневного бэкапа (МСК), формат `HH:MM` (A2). |
| `BACKUP_DIR` | `./backups` | Каталог для сохранения дампов. |
| `REMINDER_BEFORE_HOURS` | `2` | За сколько часов до визита отправлять напоминание (K4). |
| `LOG_LEVEL` | `info` | Уровень логирования: `debug`, `info`, `warn`, `error` (N6). |
| `BOT_MODE` | `polling` | Режим бота: `polling` или `webhook`. Для webhook дополнительно нужен URL и порт. |

### 1.3. Дополнительные (при использовании webhook)

| Параметр | Описание |
|----------|----------|
| `WEBHOOK_URL` | Полный URL для webhook (например `https://example.com/bot/webhook`). |
| `WEBHOOK_PORT` | Порт HTTP-сервера для приёма обновлений. |

### 1.4. Реализация конфига в коде

- Один пакет `internal/config`: загрузка из env (и при необходимости из файла `.env` или флагов).
- Валидация при старте: наличие токена, непустой список барберов, валидность `BACKUP_TIME`, `DATABASE_DSN`.
- Структура конфига не зависит от транспортного слоя (Telegram); зависимости — только стандартная библиотека и возможно `os`, `strconv`, `strings`.

## 2. Выбор базы данных

Требования: до ~200 клиентов (N1), простая архитектура, ежедневный дамп (A2), Docker Compose (A3).

### 2.1. Рекомендация: **SQLite**

- **Масштаб**: 200 клиентов и тысячи записей/визитов — нагрузка небольшая, одной БД достаточно.
- **Простота**: один файл БД, без отдельного процесса; в Docker — один контейнер приложения с volume для персистентности.
- **Бэкапы (A2)**: дамп через `sqlite3 .dump` или копирование файла; формат понятен, восстановление — загрузка дампа в новую БД.
- **Стек Go**: `database/sql` + драйвер `modernc.org/sqlite` (или `github.com/mattn/go-sqlite3` с CGO); при желании — sqlc для типобезопасных запросов.
- **Ограничения**: один процесс записи (для бота достаточно); при росте можно заменить адаптер на PostgreSQL без смены домена и use-case.

### 2.2. Альтернатива: PostgreSQL

Имеет смысл, если планируется несколько инстансов бота или отдельный сервис аналитики. Тогда с самого начала можно выбрать PostgreSQL и вынести в отдельный сервис в `docker-compose`. Для текущих требований — опционально.

### 2.3. Итог

- **Старт**: SQLite, один файл БД, WAL-режим для надёжности.
- **Порты**: интерфейсы репозиториев в core не привязаны к SQLite; при необходимости позже добавляется адаптер PostgreSQL.

## 3. План исполнения (этапы)

Архитектура: слои Entry → Use-case → Domain, порты и адаптеры по [go-backend-prompt.md](go-backend-prompt.md). Бот — единственный «entry» (Telegram long polling или webhook).

### Фаза 0: Каркас проекта и инфраструктура

| # | Задача | Результат |
|---|--------|-----------|
| 0.1 | Инициализация Go-модуля, структура каталогов (cmd, internal: domain, usecase, port, adapter) | `go.mod`, заготовки пакетов |
| 0.2 | Конфиг: загрузка env, валидация (токен, BARBER_TELEGRAM_IDS, DSN, BACKUP_TIME, LOG_LEVEL) | `internal/config` |
| 0.3 | Схема БД: таблицы (users/clients, barbers, services, visits, visit_services, schedule_settings, bans, audit/events для логов) | SQL-миграции или один init-скрипт |
| 0.4 | Адаптер репозитория: подключение к SQLite, реализация портов (минимум — сохранение/чтение для пользователей и настроек) | `adapter/repository`, работающая миграция |
| 0.5 | Docker Compose: приложение + volume для БД и backups, передача TELEGRAM_BOT_TOKEN и BARBER_TELEGRAM_IDS через env | `docker-compose.yml`, опционально `.env.example` |
| 0.6 | Логирование: slog, уровень из конфига, вывод в stdout (N6) | Единый логгер, используемый в use-case и адаптерах |

**Критерий готовности**: приложение стартует по `docker-compose up`, читает конфиг, поднимает БД, пишет лог; без бота пока можно заглушить.

### Фаза 1: Домен и базовые сценарии

| # | Задача | Требования |
|---|--------|------------|
| 1.1 | Доменные сущности: Client, Barber, Service, Visit, VisitService, ScheduleSettings, Ban | Без зависимостей от БД/Telegram |

| 1.5 | Use-case: регистрация/идентификация пользователя по Telegram ID, определение роли (клиент/барбер) | A1, разделение клиент/барбер |

**Критерий готовности**: юнит-тесты на домен и use-case с моками репозиториев; сохранение/чтение данных через адаптер.

## 5. Сводка по конфигу и БД

| Тема | Решение |
|------|---------|
| **Конфиг** | env: `TELEGRAM_BOT_TOKEN`, `BARBER_TELEGRAM_IDS` (обязательные); `DATABASE_DSN`, `BACKUP_TIME`, `BACKUP_DIR`, `REMINDER_BEFORE_HOURS`, `LOG_LEVEL`, `BOT_MODE` (опциональные с дефолтами). |
| **БД** | SQLite (рекомендуется): один файл, WAL, дамп для бэкапов; порты позволяют позже добавить PostgreSQL. |
| **План** | 4 фазы: 0 — каркас и инфраструктура, 1 — домен и репозитории, 2 — клиентский сценарий в боте, 3 — панель барбера, 4 — бэкапы и документация. |
| **Стиль кода** | KISS (N7): код максимально простой и лаконичный; без лишних абстракций и усложнений. |

После согласования плана можно переходить к Фазе 0 (создание репозитория, конфиг, схема БД, Docker).
